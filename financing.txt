## 1. Model serving setup

### GPU selection

You're on an RTX 3090 (24GB) locally. No cloud offers consumer 3090s, so the closest match is the **A10G** (24GB, same Ampere generation). Your 3-model pipeline (face detector + MVDiffusion + GS-LRM, ~7–9GB weights) fits comfortably in 24GB.

| GPU (Modal) | VRAM | ~Price/hr | Fit for you |
| --- | --- | --- | --- |
| **A10G** ✅ | 24GB | ~$1.10 | Recommended. Closest to 3090; expect ~15–20s/job (vs ~15s on 3090). |
| L4 | 24GB | ~$0.80 | Cheaper but slower memory bandwidth; diffusion likely slower. |
| A100 40GB | 40GB | ~$2.10 | Faster than 3090, but overkill/pricey for ~15s jobs. |
| T4 | 16GB | ~$0.59 | Cheapest, but 16GB is tight for 3 models + diffusion. Risky. |

### Architecture

- Wrap your GPU server in a **Modal `Cls`**: load the 3 models once per container, expose a web endpoint.
- **Drop the jobId/polling queue and threading lock.** At ~10–20s jobs with low concurrency, a synchronous HTTP call is simpler. Modal isolates each request in its own container with its own GPU, so the lock is unnecessary.
- **Keep your Next.js proxy + rate limiting** (`src/app/api/facelift/route.ts`, 5 jobs/10min/user) exactly as-is; just point `FACELIFT_URL` at the Modal endpoint.
- **Store outputs (`.ply` + turntable `.mp4`) in S3** — this is what your YC **AWS credits** are good for (they don't apply to Modal). Keeps the GPU container stateless.
- Protect the endpoint with a shared secret header (Modal `Secret`) so only your backend can call it.
- The Gemini hairstyle step stays in your app code untouched (cloud API, no GPU).

### Cold-start optimizations (the "investment")

It's mostly configuration plus structuring model loading correctly:

1. **Weights in a Modal Volume** — download `wlyu/OpenFaceLift` once; never re-pull from HuggingFace at boot.
2. **`enable_memory_snapshot=True`** — CPU memory snapshot (GA), skips repeated init work (e.g. the ~20k file reads to import torch).
3. **`experimental_options={"enable_gpu_snapshot": True}`** — GPU memory snapshot (alpha). Snapshots the GPU memory state after models are loaded, so restore skips re-reading 8GB from disk.
4. **Load all 3 models onto the GPU inside the snapshotted init** so the snapshot captures populated GPU memory + CUDA context.
5. **Deploy** (`modal deploy`) — the snapshot only materializes after ~3–5 cold boots, so don't judge timing from the first run.

```python
import modal

app = modal.App("facelift")
image = (modal.Image.debian_slim(python_version="3.11")
         .apt_install("ffmpeg", "libgl1")
         .pip_install("torch", "diffusers", "transformers", "huggingface_hub",
                      "opencv-python-headless", "numpy", "fastapi[standard]"))
weights = modal.Volume.from_name("facelift-weights", create_if_missing=True)

@app.cls(
    gpu="A10G",
    image=image,
    volumes={"/weights": weights},
    enable_memory_snapshot=True,
    experimental_options={"enable_gpu_snapshot": True},
    scaledown_window=45,      # idle seconds before scale-to-zero (cost dial)
    min_containers=0,         # on-demand; set to 1 only for demos
)
class FaceLift:
    @modal.enter(snap=True)
    def load(self):
        self.models = load_all_models("/weights")  # onto GPU

    @modal.fastapi_endpoint(method="POST")
    def process_image(self, item: dict):
        ply, video = run_pipeline(self.models, item["image"])
        # upload ply/video to S3, return URLs
        return {"ply_url": ..., "video_url": ...}
```

## 3. Scale-to-idle economics & user scenarios

<aside>
💸

**A warm GPU bills at the full per-second rate** whether it's running inference or sitting idle. There is no idle discount. So `scaledown_window` (how long a container stays alive after the last request) is literally "how many idle GPU-seconds am I willing to buy to avoid a cold start."

</aside>

### Per-generation GPU cost components (A10G @ $0.000306/s)

| Component | Cost |
| --- | --- |
| GPU active (~18s) | ~$0.0055 |
| Cold start (~12s, isolated only) | ~$0.0037 |
| Idle tail @ 90s window | ~$0.0275 |
| Idle tail @ 30s window | ~$0.0092 |

### User scenarios (GPU cost only — add Gemini separately)

- **Bursty / session (likely):** a user fires several edits, then views results. One cold start + one idle tail amortized across many gens → GPU ≈ **~$0.01/gen**.
- **Isolated (worst case):** every generation stands alone, each eats a full idle tail → GPU ≈ **$0.037/gen @ 90s**, **~$0.019/gen @ 30s**.

<aside>
🎛️

**Recommendation: 30–45s window + fast cold starts.** Your jobs already take ~15–20s, so a ~12s cold start on an isolated return is barely noticeable. This captures rapid back-to-back edits (warm) while scaling down during multi-minute viewing gaps — cutting worst-case GPU idle cost ~3×. Flip `min_containers=1` only on demo days.

</aside>

## 5. Gemini image-edit model options

The hair edit is your quality-defining step, and the model choice is also your biggest cost lever. Current Google options (mid-2026):

| Model | ~Cost/image | Best for |
| --- | --- | --- |
| **Gemini 2.5 Flash Image** ("Nano Banana") | $0.039 (≤1024px) | **Default.** Fast, strong identity preservation + edit quality. Run all exploration gens here. |
| Gemini 3.1 Flash Image | ~$0.07 | Newer fast model; quality/reasoning lift. A/B test vs 2.5 at your output resolution. |
| **Gemini 3 Pro Image** ("Nano Banana Pro") | $0.134 (1K/2K), $0.24 (4K) | **Premium final / HD export only.** Best fidelity, detail, legible text. Too costly per-gen for exploration. |
| Imagen + mask inpainting (Vertex) | ~$0.04 | Pixel-locks the face: edits only the masked hair region. Strongest identity guarantee. |

### Which to use for good-looking renders

- **Conversational models (Gemini Flash/Pro)** take image + prompt and regenerate the whole image. Excellent realistic hair and good identity, but can subtly drift the face/skin across edits.
- **Mask-based inpainting (Imagen)** edits only the masked hair/scalp pixels and leaves the face untouched — the strongest guarantee the face doesn't change between styles. You already run a face detector + SMIRK/FLAME alignment, so you can derive a hair mask and feed it in. More integration work, maximum fidelity.

**Recommended setup:**

- **Default:** Gemini 2.5 Flash Image ($0.039) for every exploration generation — keeps all tiers at 55–62% margin.
- **Quality test:** A/B Gemini 3.1 Flash Image (~$0.07). At that cost the $14.99/100 tier compute rises to ~$7 (~46% margin) and $2/10 to ~$0.70 — still viable, so adopt broadly only if the quality lift is visible.
- **Premium "money shot":** offer Nano Banana Pro for the single final HD render the user takes to the barber — reserved to $14.99 or sold as a small HD add-on.
- **If testers report face drift:** pilot Imagen mask-inpainting using your existing hair mask.
- **Batch API (50% off, async 12–24h):** not for realtime, but halves cost for offline jobs like pre-baking your free-tier hair `.ply` library.

</aside>
